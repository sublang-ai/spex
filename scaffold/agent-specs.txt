## Specs (Source of Truth)

- Canonical specs live in @specs/.
  Start with @specs/map.md for a quick-reference index of all spec files and @specs/meta.md for the spec format.
- Before making changes or suggestions in an area that has a spec, read the relevant spec file(s) first and follow them.
- Whenever your changes involve any spec, or you make a decision worth recording, keep the specs in sync — don't let them drift.
- Keep specs:
  - Complete & coherent: sufficient for you to reimplement code.
  - Right level: each package file in @specs/packages holds user-visible behavior (## External Behavior), implementation requirements (## Internal Behavior), and tests of its own claims (## Verification); behavior and tests spanning multiple packages go in @specs/compositions. Specify integration/system testing, not unit testing.
  - Minimal: essential and concise; also check against other spec items to avoid overlap.
  - Well organized: spec packages are finely scoped, with high cohesion and low coupling.
- Run `spex lint` to check structure, item IDs, and citations after editing specs.
