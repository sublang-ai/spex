## Specs (Source of Truth)

- Canonical specs live in @specs/.
  Start with @specs/map.md for a quick-reference index of all spec files and @specs/meta.md for the spec format.
- Before making changes or suggestions in an area that has a spec, read the relevant spec file(s) first and follow them.
- Whenever your changes involve any spec, or you make a decision worth recording, keep the specs in sync — don't let them drift.
- Keep specs:
  - Complete & coherent: sufficient for you to reimplement code.
  - Right level: each file in @specs/packages is a standalone contract using the applicable ## External Behavior, ## Internal Behavior, and ## Verification sections. External is offered to any human, host, component, or higher-layer package user; Internal is hidden semantic behavior, not implementation detail. An exact peer External citation may express an intentional fixed semantic dependency. Installed assembly roles, supplier selections, and integrated scenarios go in @specs/compositions; package files contain no installation annotations. Put direct composition and verification citations inline beside the phrase or assertion they qualify; use no detached `Composes:`, `Bindings:`, or `Verifies:` lines. Specify contract, binding-conformance, integration, and acceptance verification, not unit tests.
  - Minimal: essential and concise; also check against other spec items to avoid overlap.
  - Well organized: spec packages are finely scoped, with high cohesion and low coupling.
- Validate structure, item IDs, exact links, and inline citation rules with format-compatible checks after editing specs; never restore removed trace lines to satisfy an older checker.
